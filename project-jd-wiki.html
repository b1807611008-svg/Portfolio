Validation Loop

我是如何跑通 Relay 设计稿到 Design Wiki 的闭环流程。

01

设计输入

我先选择视频场域下的真实业务组件作为测试对象，而不是使用孤立 demo。输入范围覆盖 token、atom、component、module 到 page，让流程从一开始就面对真实复杂度。

  • Relay 设计稿
  • 颜色、字号、间距等 token
  • 卡片、导航、表单等组件
02

连接层

通过 Relay MCP 读取节点结构、样式 token、组件层级、页面 frame 和必要的截图坐标信息，让 AI 理解设计稿结构，而不是只看一张视觉截图。

  • 读取节点结构
  • 读取组件层级
  • 补充截图与坐标语义
03

Skill 生成

基于已有 relay-to-design.md skill，我连续跑多个视频场域组件，生成 design.md，并观察文档中的 meta、tokens、atoms、components、modules、layout、interactions 是否稳定。

  • 解析设计稿
  • 生成 design.md
  • 检查结构字段完整度
04

产物输出

我和带教、+1 一起沉淀文件职责：design.md 描述设计源文件，spec.md 面向实现规范，behaviors.md 描述交互状态，variants.md 记录不同状态变体，assets/ 管理图片和图标资源。

  • design.md
  • spec.md / variants.md / behaviors.md
  • ai-schema.yaml / CHANGELOG.md
05

UI 还原

我将 design.md 进一步还原为 HTML 预览，用页面效果反向检查结构化文档是否足够准确。这个环节帮助我发现单看文档不容易暴露的问题。

  • design.md 转 HTML
  • 生成 spec-page.html
  • 还原门户页面预览
06

多维验证

我从 schema、视觉还原和语义三个维度检查结果，记录 icon 还原度低、token 映射不完整、文档结构不稳定、HTML 渲染不符合预期等问题。

  • Schema Validation
  • Visual Validation
  • Semantic Validation
07

闭环优化

没有通过的内容会回到连接层、skill 或文档结构中继续调整。我将问题整理反馈给许帅，部分内容也通过 PR 参与到 skill 的优化迭代中。

  • 回写 design.md 或模块内容
  • 更新设计稿模块
  • 重新生成并迭代验证
价值总结 结构化 可验证 可追溯 可复用 高效协作

Relay Design Design Wiki Engineering
6+ 核心文件类型
MD Source of Design

design.md

描述从 Relay 设计稿中抽取出来的设计事实，包括页面结构、模块层级、布局关系、设计 token、组件组成和视觉语义。

用来回答：设计稿里到底有什么？
SP Implementation Spec

spec.md

把设计信息转译成开发可执行的实现规范，明确尺寸、间距、颜色、字体、组件规则和页面还原要求。

用来回答：开发应该如何实现？
BH Interaction Rules

behaviors.md

记录组件或页面的交互行为，例如 hover、click、loading、empty、error、disabled 等状态下的响应方式。

用来回答：用户操作后会发生什么？
VR State Variants

variants.md

沉淀组件或模块的不同形态与状态变体，帮助后续复用、扩展和对齐多场景下的视觉差异。

用来回答：这个组件有哪些版本？
AI Machine Schema

ai-schema.yaml

提供给 AI 或自动化流程读取的结构化 schema，用字段化方式描述组件、token、模块和校验规则。

用来回答：机器如何稳定理解这份设计？
AS Visual Assets

assets/

集中管理从设计稿导出的图片、图标、截图和其他静态资源，保证 HTML 预览与实现过程有可追溯的素材来源。

用来回答：图片和图标资源在哪里？
CL Change Record

CHANGELOG.md

记录每轮生成、人工修正、验证反馈和结构调整，让设计文档的演进过程可以被复盘和追踪。

用来回答：这份文档经历了哪些变化？

What I Did

把一个仍在迭代的 AI skill，放进真实业务组件里反复验证。

在 Design Wiki 项目中，我主要参与设计资产结构化流程的验证、落地和推广。项目里的核心 skill 由正职同学提前搭建基础能力，我的重点不是从零开发工具，而是基于已有能力，在真实业务场景中系统测试它是否可用、稳定、可推广。

我选择视频场域下的一系列组件作为测试对象，从基础 token 开始，到组件文档生成、页面级 Design.md 组织，再到 HTML 预览渲染，完整跑通了从 token、组件到页面的链路，验证 Design Wiki 方向的可行性。

Documentation System

让设计资产进入一个可被理解、检索和复用的文件结构。

01

Foundations

基础规范沉淀 token、颜色、字号、间距等设计变量，保证后续组件生成有明确的源头。

02

Horizontal / Components

组件层沉淀 spec、variants、behaviors 与 ai-schema，让不同状态和交互语义被结构化表达。

03

Pages

页面级内容通过 Design.md 串联模块、布局、语义和验证目标，并支持 HTML 预览还原。

Contribution

我的贡献不是单点输出，而是把验证经验变成团队可以复用的方法。

跑通链路

基于真实业务组件，连续验证 token、组件、页面到 HTML 预览的完整流程。

暴露问题

在多组件测试中发现 icon 资产还原度低、token 映射不完整、文档结构不稳定、HTML 渲染不符合预期等问题。

推动迭代

整理问题并寻找解决方案，将优化点反馈给许帅，部分内容通过 PR 参与到 skill 迭代中。

沉淀 SOP

输出《将 Relay 设计稿转化成 Design.md 的标准化流程》，帮助设计同学理解文件职责、检查项和协作方式。

Meaning

这件事的价值，是把 AI 工具能力和真实设计业务连接起来。

  1. 项目层验证 Design Wiki 方向在真实组件场景中的可行性。
  2. 工具层在正式推广前发现 skill 在复杂场景里的优化点。
  3. 规范层把个人验证经验转化成可复用的目录和文档结构。
  4. 协作层让设计、开发、AI 在同一套结构化资产里工作。
  5. 个人层理解 AI 不是替代设计判断，而是放大结构化沉淀和团队协作的效率。
