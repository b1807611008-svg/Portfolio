01FOUNDATION

Token

颜色、字体、间距等基础规则。作为唯一事实来源，为所有上层视觉能力提供统一变量。

color.primary: #2F8EE8
space.md: 16px
radius.card: 12px
不描述具体组件
02PRIMITIVES

Visual

图标、蒙层、文字等最小视觉能力。引用 Token，将抽象变量转换成可组合的视觉单元。

visual: title-primary
font: {type.heading}
color: {color.text-1}
不承担业务结构
03COMPOSITION

Component

描述组件结构、变体、状态与交互，明确可以如何使用，以及在什么边界内使用。

variant: compact | default
state: loading | empty
behavior: tap → detail
不重复基础样式
04ORCHESTRATION

Page

调用并编排已有组件，描述页面结构、顺序和业务上下文，不重复解释组件内部规则。

page: channel-home
uses: Header, FeedCard
layout: vertical-flow
只描述组合关系
APRIVATE

Assets

存放组件私有图片与图标资源，资源跟随组件归档，避免一次性素材进入并污染全局资产库。

components/feed-card/
├─ design.md
└─ assets/cover.webp
谁使用，谁管理
CORE PRINCIPLE核心原则

每一层只描述自己该负责的内容。

前面的规则越准确，AI 在页面层需要理解和猜测的内容就越少。
规则准确度↑AI 猜测成本↓

02

连接层

通过 Relay MCP 读取节点结构、样式 token、组件层级、页面 frame 和必要的截图坐标信息，让 AI 理解设计稿结构，而不是只看一张视觉截图。

  • 读取节点结构
  • 读取组件层级
  • 补充截图与坐标语义
03

Design Outline 预检

正式生成文档前，先让 AI 输出 Design Outline，预检页面层级、组件边界、复用关系与缺失信息。通过人工确认提纲，避免错误结构直接进入最终文档。

  • 输出页面与组件提纲
  • 确认层级和复用边界
  • 补齐缺失信息后再生成
04

生产组件化文档包

Outline 通过后，按组件边界批量生成结构化文档包。每个组件独立管理设计事实、实现规范、变体、行为与私有资源，让文档可以被检索、调用和持续维护。

  • design.md / spec.md
  • variants.md / behaviors.md
  • ai-schema.yaml / assets/ / CHANGELOG.md
05

验证：检验规范是否可执行

让 AI 仅依据组件文档还原 HTML 或完整页面，用实际渲染结果反向检验规范是否足够明确、完整且可执行，暴露单看文档难以发现的问题。

  • 文档驱动 HTML / 页面还原
  • 检查结构与视觉一致性
  • 验证状态和交互语义
06

沉淀：进入 Design Wiki

根据验证结果修正 Token 映射、组件边界、状态与交互规则，更新对应文档和版本记录。通过验证的组件包最终进入 Design Wiki，成为团队和 AI 可复用的设计资产。

  • 修正规则并回写文档
  • 更新 CHANGELOG 与版本
  • 发布至 Design Wiki

6+ 核心文件类型
MD Source of Design

design.md

描述从 Relay 设计稿中抽取出来的设计事实，包括页面结构、模块层级、布局关系、设计 token、组件组成和视觉语义。

用来回答：设计稿里到底有什么？
SP Implementation Spec

spec.md

把设计信息转译成开发可执行的实现规范，明确尺寸、间距、颜色、字体、组件规则和页面还原要求。

用来回答：开发应该如何实现？
BH Interaction Rules

behaviors.md

记录组件或页面的交互行为，例如 hover、click、loading、empty、error、disabled 等状态下的响应方式。

用来回答：用户操作后会发生什么？
VR State Variants

variants.md

沉淀组件或模块的不同形态与状态变体，帮助后续复用、扩展和对齐多场景下的视觉差异。

用来回答：这个组件有哪些版本？
AI Machine Schema

ai-schema.yaml

提供给 AI 或自动化流程读取的结构化 schema，用字段化方式描述组件、token、模块和校验规则。

用来回答：机器如何稳定理解这份设计？
AS Visual Assets

assets/

集中管理从设计稿导出的图片、图标、截图和其他静态资源，保证 HTML 预览与实现过程有可追溯的素材来源。

用来回答：图片和图标资源在哪里？
CL Change Record

CHANGELOG.md

记录每轮生成、人工修正、验证反馈和结构调整，让设计文档的演进过程可以被复盘和追踪。

用来回答：这份文档经历了哪些变化？

What I Did

把一个仍在迭代的 AI skill，放进真实业务组件里反复验证。

在 Design Wiki 项目中，我主要参与设计资产结构化流程的验证、落地和推广。项目里的核心 skill 由正职同学提前搭建基础能力，我的重点不是从零开发工具，而是基于已有能力，在真实业务场景中系统测试它是否可用、稳定、可推广。

我选择视频场域下的一系列组件作为测试对象，从基础 token 开始，到组件文档生成、页面级 Design.md 组织，再到 HTML 预览渲染，完整跑通了从 token、组件到页面的链路，验证 Design Wiki 方向的可行性。

Independent Practice / Skill Lab

从“可阅读”到“可调用”

让组件文档真正成为AI 的设计能力

我的目标不只是把设计成果换成文档供人查阅。否则，它仍然只是另一种形式的设计说明，并没有充分发挥 AI 可读资产的价值。

因此，我继续沿着组件库生产与调用链路，独立实践了三个 Skill：先提升 16.0 组件库的生产效率，再尝试让 AI 结合 PRD、组件文档与 Skill，识别需求结构、调用已有组件并生成设计稿。

01ANNOTATION SKILL

组件自动标注

在组件整理过程中，我发现间距标注存在大量重复劳动。不同标注对象的颜色、优先级和伸缩行为都需要设计师反复判断与设置。

我将这些判断规则沉淀为标注 Skill，让 AI 自动识别对象关系并完成一致的规范标注。

输入
待标注组件与图层结构
规则
标注颜色 · 对象优先级 · 伸缩规则
输出
自动生成的规范标注
减少重复操作统一标注判断提升整理效率
